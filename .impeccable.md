# Design Context — Sodexo Digital & AI Experience Catalogue

This file is the source of truth for design decisions on this codebase. It implements the [impeccable](https://github.com/pbakaus/impeccable) skill's `teach` output plus ui-ux-pro-max priorities. Read this before building or reviewing any UI.

Related docs: `PRODUCT_VISION.md` (strategy), `.cursor/rules/sodexo-experience-catalogue.mdc` (stack/naming).

---

## 1. Brand posture

Three words for the interface personality:

> **corporate-trusted · optimistic · humane**

- **Corporate-trusted** — this is a Sodexo-branded Innovation artefact shown to business developers and eventually clients. Credibility > novelty.
- **Optimistic** — catalogue of *experiences* (not products) for consumers in WORK/LEARN/HEAL/PLAY. The tone should feel hopeful and human-centred, never clinical or salesy.
- **Humane** — journey-first, persona-first. Every screen starts from a real person's day.

## 2. Target audience

- **Phase 1–3**: internal Sodexo (business developers, account managers, Innovation leaders). Desktop-heavy, English-first, analytical.
- **Phase 4**: Sodexo clients. Desktop + mobile, multi-language (FR first, then ES/DE/PT).

Design for the phase-1 user first, but do not make choices that would block phase-4 trust (no emoji, no jokey copy, no untranslatable idioms).

## 3. Quality bar

**Flagship.** Per `PRODUCT_VISION.md` OKR2: "a visually excellent experience that reflects Sodexo's Digital & AI brand". MVP polish is not acceptable; this is a showcase.

## 4. Typography

### Current family (as of 2026-04-20)

- **Everything**: `Open Sans` (Google Fonts, humanist sans) — used for both display and body.

Loaded exclusively via `next/font/google` in `src/app/layout.tsx`; never via `@import url(...)` — FOIT + CLS guaranteed.

CSS variables:
- `--font-heading` → Open Sans stack (kept as a distinct token so a display swap can happen later without touching components)
- `--font-body` → Open Sans stack

### Rules

- Body minimum 16px (1rem). Never smaller for paragraph text.
- Line height: 1.1–1.2 headings; 1.5–1.7 body.
- Use the scale defined in `src/styles/tokens.css` (`--text-xs` → `--text-5xl`). Do not invent one-off `clamp()` values inside components.
- Weights: 400 body, 500 labels, 600 subheadings, 700–800 headings.
- One display font + one body font; do not add a third.
- Tabular numerals for KPIs: `font-variant-numeric: tabular-nums` when the value is data.

### Banned (reflex choices)

- `Inter`, `Roboto`, `Noto Sans`, `Lato`, `Nunito` as primary display or body.
- Any two sans fonts that read optically identical.
- Arbitrary inline font sizes not on the scale.

## 5. Color

Sodexo brand blue drives the whole system. The palette lives in `src/styles/tokens.css`.

### Principles

- **60-30-10**: 60% surface (`--surface`), 30% brand blue (`--blue`, `--blue-solid`), 10% accent (`--blue-primary` or area colour).
- Tint neutrals toward the brand hue. `--grey-subtle` and `--grey-border` are already cool-biased; do not introduce pure `#000`/`#888` greys.
- Area tints (`--area-work`, `--area-learn`, `--area-heal`, `--area-play`) are reserved for the area context they name. Do not use PLAY blue on a HEAL screen.
- All foreground/background pairs must meet WCAG AA: **4.5:1** for body text, **3:1** for UI elements and large text.
  - Known current violation: `--teal` (#00d1c7) on white is ~2.4:1. Use `--teal-dark` (#00a096) when teal must appear as text.
- Status colours (`#27ae60` scaled, `#e67e22` scaling, `#3498db` pilot) are functional; always pair with an icon or label, never colour-only.

### Banned

- **Purple → blue gradient** as a hero background (AI-slop signature). Use a solid brand colour or a single-hue duotone.
- **Gradient text** (`background-clip: text`).
- White at <80% opacity on any coloured surface that is not near-black (`<20%` luminance) — contrast fails.
- Raw hex values inside component JSX when a token exists. Prefer `var(--token)` or a Tailwind class that maps to a token.

## 6. Layout & spacing

- 4pt / 8pt spacing scale from `--space-2xs` (4px) → `--space-xl` (40px). No arbitrary pixel values.
- Radii from `--radius-xs` (4px) → `--radius-xl` (25px) + `--radius-pill`. Retire one-off values (`rounded-[20px]`, `rounded-[28px]`, `rounded-[40px]`) — use the token ladder.
- Shadows from `--shadow-sm` → `--shadow-panel` + `--shadow-nav` + `--shadow-benefits`. Never write a new `shadow-[0_...]` inline unless there is no matching token; if needed, add the token first.
- **Max 2 levels of card nesting.** A card inside a card is acceptable with intent. A card inside a card inside a card is not.
- Container max-width 1280–1600px. Every top-level page has a `max-w-*` constraint.
- Touch targets ≥ 44×44px on any interactive element. Use `hitSlop`-equivalent padding if the visual target must be smaller.

### Banned

- Side-stripe borders (`border-left > 1px` on cards).
- Fixed widths (`w-64`, `w-72`) without a `max-width` and a responsive fallback.
- Glassmorphism (`backdrop-filter: blur(...)`) as a default decoration. Reserved for: modal/sheet dismiss affordance, nav bar over scrolled content.

## 7. Motion

- **Library**: `framer-motion` (to be installed in Phase 2).
- **Easing**: exponential ease-out `cubic-bezier(0.25, 1, 0.5, 1)` for entrances. `cubic-bezier(0.5, 0, 0.75, 0)` for exits. Never `linear`, `bounce`, or `elastic`.
- **Durations**: 100–150ms feedback, 200–300ms state changes, 500–800ms entrances. Nothing over 800ms on UI.
- **What to animate**: `transform` + `opacity` only. Never `width`, `height`, `top`, `left`, `margin`.
- **Reduced motion**: every animation must have a `prefers-reduced-motion: reduce` branch that either removes or drastically shortens it. Test with the OS setting enabled.
- **Stagger**: 30–50ms between siblings for list reveals. Cap at ~8 items.
- **Budget**: max 2 animations per viewport at any time. Restraint > activity.

### Banned

- Decorative animations that don't convey a cause-effect relationship.
- Animations that block user input (buttons unclickable during transition).
- Parallax that moves during scroll without a reduced-motion escape.

## 8. Icons

- **Icon system**: `@phosphor-icons/react` (single source). Stroke/fill style by hierarchy:
  - `weight="bold"` or `weight="fill"` for primary UI (nav, CTAs, section headers).
  - `weight="regular"` for secondary chrome.
  - Never mix `duotone` with the others in the same cluster.
- Sizes on a scale: 14 / 16 / 18 / 20 / 24 / 32 / 40.
- **No emoji as UI icons.** Ever. Not for module tiles, not for solution hero art, not for status. The Notion CMS provides an `img` field that today stores an emoji — when piped to the UI, it must be transformed to a Phosphor icon or a mapped SVG.
- Flags (country flags on solution cards) are acceptable as emoji *if* they carry an `aria-label` with the country name and a text fallback is provided for screen readers.

## 9. Content (UX writing)

- Button labels: verb-first, concrete. "See 4 solutions" > "View more".
- Error messages: state cause + recovery. "Password incorrect — if you need access, email innovation@sodexo.com." > "Incorrect password.".
- Empty states: always offer a next action.
- Headings: sentence case, not Title Case, and never ALL CAPS except for eyebrow labels ≤ 20 chars.
- Data labels: noun phrases. "Solutions available" > "Available solutions".

## 10. Accessibility floor

Non-negotiable:

- Skip link on every page (`Skip to main content` at keyboard-focus).
- Heading hierarchy: h1 → h2 → h3 with no skips. Check per route.
- Every `<input>` has a `<label>` with `for=`, not placeholder-only.
- Every icon-only button has `aria-label`.
- Every meaningful image has `alt=`; decorative images have `alt=""` + `aria-hidden`.
- `prefers-reduced-motion` respected.
- All interactive elements have a visible `:focus-visible` ring (never `outline: none` without replacement).

## 11. Quality checkpoint (ask before shipping)

> "If someone saw this, would they instantly recognize it as AI-generated?"

If the answer is "maybe yes" for any of these reasons — refine before merging:

- Purple-blue gradient hero ✗
- 3+ identical rounded cards in a grid ✗
- Open Sans / Inter anywhere ✗
- Emoji where a Phosphor icon belongs ✗
- More than 2 levels of card nesting ✗
- Shadow values not drawn from the token set ✗
- Any `clamp()` that isn't a typography ramp ✗

## 12. Anti-patterns specific to this project

- **"25 modules on the persona page"** — showing every module regardless of whether it applies to the persona's journey. Fix: filter to modules that have at least one moment overlap with the persona's journey steps.
- **"Emoji-as-icon leak"** — Notion `img` field surfaces emoji into production UI. Fix: central mapping table `emoji → Phosphor icon`, fallback to a generic module icon.
- **"Navbar-without-persona-context"** — once a user clicks through to a module or solution, the persona context evaporates. Fix: keep a persistent persona chip in the breadcrumb/navbar.
- **"Fake listen button"** — the moment overlay has a "Listen" button that does nothing. Fix: either wire text-to-speech (Web Speech API) or remove the button.
